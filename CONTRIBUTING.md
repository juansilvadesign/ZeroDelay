# Contributing to ZeroDelay

Thanks for your interest in contributing! This guide explains how to run the
extension locally, the code style we expect, and how to submit a good Pull
Request.

ZeroDelay is free software under the **GPL-3.0-or-later** license. By
contributing, you agree that your code will be distributed under that same
license.

## Requirements

- **Node.js 20+** (CI uses version 22) and npm — needed for lint, tests and
  build.
- **Google Chrome** (or another Chromium) and/or **Firefox** to test.

Install the development dependencies:

```bash
npm install
```

## Running the extension locally

### Chrome / Chromium

1. Go to `chrome://extensions`.
2. Turn on **Developer mode** (top-right corner).
3. Click **Load unpacked** and select the **repository root folder** (where
   `manifest.json` is).
4. Open a YouTube live stream with DVR enabled to test. After changing files,
   click **Reload** on the extensions page and reload the YouTube tab.

### Firefox

Firefox uses its own manifest, generated by the build. Use `web-ext`:

```bash
npm run run:firefox
```

This builds the Firefox version in `dist/firefox` and opens a temporary browser
instance with the extension already loaded.

### Zen Browser

Zen Browser uses the same Firefox/Gecko build:

```bash
npm run run:zen
```

If the binary isn't on your `PATH`, point to it manually:

```bash
ZEN_BINARY=/opt/zen-browser-bin/zen-bin npm run run:zen
```

To keep a test profile between runs, set
`ZEN_PROFILE=/path/to/profile`.

## Checks before opening a PR

Run the same three checks that CI runs:

```bash
npm run lint      # ESLint — style and correctness
npm test          # unit tests (node --test)
npm run build     # validates and packages the extension into build/
```

All must pass (exit 0). For the Firefox/AMO-specific publishing validation, use
`npm run lint:firefox`.

## Code style

- Style is enforced by **ESLint** (`eslint.config.mjs`). Run `npm run lint` and
  fix the warnings before submitting.
- Write code in the same language and style as the surrounding files. Comments
  and user-facing text stay in **Portuguese (pt-BR)**; UI strings must be
  localized in `_locales/en` and `_locales/pt_BR` (keep both in sync).
- Avoid new runtime dependencies: the extension runs without a bundler and
  without a build step in the browser.

## ⚠️ Important constraint: storage keys

The extension persists settings in `chrome.storage.local`. **Renaming or removing
an existing key breaks users' saved settings.** In particular, the key
`skipThreathold` (with that historical spelling) **cannot be renamed**. When
adding settings, create new keys instead of changing existing ones.

## Testing on a real live stream

Before opening a PR, test the behavior on a **real live stream** on YouTube (not
a regular video), preferably one with DVR/delay, and check that:

- Latency catch-up works (the speed rises when you're behind and returns to
  `1.0x` once you reach live);
- The player indicators show correctly when enabled;
- Nothing breaks in fullscreen or in the embedded player.

State in the PR description which browser(s) you tested.

## Commit conventions

- Write clear, focused messages in the imperative (e.g. "Fix...", "Add...").
- A concise subject on the first line; details in the body if needed.
- One logical change per PR whenever possible — small PRs get reviewed faster.
- Reference the related issue (e.g. `Closes #3`).

## Opening the Pull Request

1. Fork and create a branch from `main`.
2. Make your changes and run the checks above.
3. Open the PR and fill in the template — especially the storage and testing
   checklist.
4. Update [CHANGELOG.md](CHANGELOG.md) in the *Unreleased* section when the
   change matters to users.
